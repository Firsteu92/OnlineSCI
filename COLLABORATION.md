# Collaboration Guidelines — 协作制度

本文档是 OnlineSCI 项目的协作总纲。所有参与者在加入前请仔细阅读。

---

## 1. 项目性质

OnlineSCI 是一套标准化 AI 辅助科研工作流工具包，也是一个开放的研究协作框架。参与者可以在空白项目位（P01–P10）上启动自己的研究项目，利用 AI 进行文献调研、数据分析、图表生成和论文撰写，人类科学家负责审核和纠偏。研究过程全程公开记录。

---

## 2. 角色定义

### 2.1 项目负责人 / Project Lead
- 负责项目整体方向、里程碑推进和最终决策
- 拥有论文署名顺序的最终裁定权（在充分协商的基础上）
- 管理 GitHub 仓库的 merge 权限

### 2.2 Prompt Engineer / AI 交互负责人
- 将科研目标拆解为 AI 可执行的指令链
- 负责数据预处理、AI 代码生成、初稿框架搭建
- 记录所有关键的 AI 对话日志

### 2.3 Domain Expert / 领域审查人
- 在关键节点进行专业判断
- 重点关注：方法合理性、结果解读、因果推断
- 不要求每天参与，但在关键评审节点必须响应

### 2.4 Code Reviewer / 代码审查员
- 验证 AI 生成的代码是否存在逻辑漏洞
- 确保代码可运行、可复现
- 审查数据处理流程的正确性

### 2.5 Literature Scout / 文献侦察兵
- 跟踪前沿文献，确认选题新颖性
- 对 AI 生成的每一条参考文献进行真实性核验
- 找到合适的对标论文

> 一个人可以同时担任多个角色。角色可以在项目进展中动态调整。

## 3. 参与者要求与加入方式

### 3.1 基本要求

| 要求 | 说明 |
|------|------|
| 学科背景 | 自然科学相关专业。非自然科学背景的参与者若能证明具备相关知识储备，同样欢迎 |
| 学位要求 | 不设学位门槛 |
| 英文能力 | 能阅读和理解英文学术论文 |
| 工具能力 | 能使用 GitHub 进行基本的协作操作 |
| 时间投入 | 不要求全职参与，但需要在关键节点及时响应 |

### 3.2 如何加入

1. 阅读本文档和 README.md，了解项目性质和协作规则
2. 选择感兴趣的项目，查看 projects/DASHBOARD.md
3. 通过 GitHub Issue 联系，标题格式：`[加入] 你的姓名 - 感兴趣的项目编号`
4. 项目负责人确认后，在 CONTRIBUTORS.md 中登记

## 4. 署名制度

### 4.1 署名标准

遵循 [CRediT（Contributor Roles Taxonomy）](https://credit.niso.org/) 国际学术贡献分类标准。署名要求满足以下至少两项：

| CRediT 维度 | 说明 |
|---|---|
| Conceptualization | 提出核心科学问题或研究假设 |
| Methodology | 设计研究方案或分析方法 |
| Software | 编写或审查关键代码 |
| Validation | 结果验证与纠错 |
| Formal Analysis | 数据分析与解读 |
| Data Curation | 提供或整理关键数据 |
| Writing – Original Draft | 撰写论文初稿 |
| Writing – Review & Editing | 实质性修改论文内容 |
| Visualization | 制作关键科学图表 |
| Supervision | 项目监督与方向把控 |

### 4.2 署名顺位

第一作者授予对论文贡献最大的参与者。若两人贡献难以区分，可设共同第一作者（Co-first author）。

通讯作者是论文投稿、审稿回复和学术质量的最终责任人，通常为项目发起人或提供核心未发表数据的 PI。

满足署名标准且有实质性 GitHub 贡献记录的参与者列为共同作者。

### 4.3 署名协商时间节点

- **S0 阶段结束时**：初步确定拟定作者列表和预期顺序
- **S2 阶段结束时**：根据实际贡献调整作者列表
- **S8 投稿前**：最终确认署名顺序

### 4.4 AI 的署名问题

AI（Claude、GPT 等）不列为作者。AI 的贡献将在 Methods 部分如实声明。

## 5. 项目阶段与工作流

OnlineSCI 使用 S0–S8 标准化阶段：

| 阶段 | 内容 |
|------|------|
| S0 | 项目建立 |
| S1 | 文献与选题核验 |
| S2 | 实验协议冻结 |
| S3 | 数据准备与探索 |
| S4 | 正式实验 |
| S5 | 验证与证据整理 |
| S6 | 论文逻辑与图件 |
| S7 | 写作与内部审查 |
| S8 | 投稿与归档 |

每阶段有对应的 G0–G5 质量门槛。

---

## 6. AI 使用规范

- 所有关键 AI 对话必须保存日志到各项目的 `logs/` 目录
- 记录格式：日期 + 模型 + 任务描述 + 关键输出摘要
- 投稿前汇总 AI 参与度报告
- **文献幻觉零容忍**：每篇引用必须人工核验
- AI 生成的所有代码必须能在标准环境中复现
- 敏感数据不得直接粘贴到公共 AI 对话界面

---

## 7. 数据管理

- 仓库不存储原始数据，仅保存数据链接和脚本
- 每个项目的 `data/` 目录包含 Manifest 和校验信息
- 大型数据存储在本地、服务器或合规云存储中
- 数据文件不上传到公开仓库

---

## 8. 知识产权

- **文档和模板**：CC BY-NC-SA 4.0
- **代码**：当前统一使用 CC BY-NC-SA 4.0，将来可能改为 MIT 或 Apache-2.0
- **未发表数据**：不放入公开仓库，所有权归数据提供方
- **论文草稿**：投稿前不公开。投稿后按目标期刊的版权政策处理
- Git commit 记录为所有贡献提供时间优先权证明

---

## 9. 退出机制

参与者可以在项目任何阶段退出。不同阶段退出的署名处理：

| 退出时间 | 署名处理 |
|---|---|
| S0（项目建立） | 不署名，致谢 |
| S1–S2（文献/协议） | 根据实际贡献判断 |
| S3–S5（实验/验证） | 保留署名权，顺序调整 |
| S6–S8（写作/投稿） | 署名不变 |

退出者保留其已完成贡献的署名权，需将未完成工作交接。

---

## 10. 投稿策略

采用三级投稿策略：

| 级别 | 说明 |
|---|---|
| T1（首投） | 领域顶刊 |
| T2（改投） | 高水平综合/专业期刊 |
| T3（保底） | 领域主流 SCI 期刊 |

拒稿后分析审稿意见，按级别依次改投。三次被拒后由作者集体讨论决定是否继续。

---

## 11. 争议解决

1. 技术争议：在 GitHub Issue 中公开讨论，以证据和数据为准
2. 署名争议：全体参与者协商，项目负责人最终裁定
3. 如无法达成一致，可邀请项目外的资深学者作为中立仲裁
